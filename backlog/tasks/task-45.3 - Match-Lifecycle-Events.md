## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Turn events (TASK-45.2) cover what happened during gameplay, but the full
lifecycle story starts before the first action. A complete match log must include:

- `match.created` — the match was created with specific parameters
- `player.joined` — each player (human or bot) connected to the match
- `game.initialized` — the `system:init` action was applied and the first state
  hash was recorded
- `game.completed` — the match reached `gameOver` with outcome, final LP values,
  turn count, and wall-clock duration

These events are server-level, not engine-turn-level. They are generated by the
server at key transition points in the match lifecycle and prepended to the
turn-derived events to form the unified `MatchEventLog`.

### MatchEventLog structure

```typescript
interface MatchEventLog {
  matchId: string;
  events: PhalanxEvent[];   // lifecycle + all turn events in order
  fingerprint: string;      // SHA-256 of JSON.stringify(events) with sorted keys
  generatedAt: string;      // ISO timestamp
}
```

The `fingerprint` is computed from the full ordered event array. Storing it
alongside the events allows offline verification without re-deriving.

1. Add `MatchEventLogSchema` to `shared/src/schema.ts`:
   ```typescript
   export const MatchEventLogSchema = z.object({
     matchId: z.string().uuid(),
     events: z.array(PhalanxEventSchema),
     fingerprint: z.string(),
     generatedAt: z.string().datetime(),
   });
   ```

2. Add lifecycle event name constants to `shared/src/telemetry.ts`:
   `MATCH_CREATED`, `PLAYER_JOINED`, `GAME_INITIALIZED`, `GAME_COMPLETED`.

3. In `server/src/match.ts`:
   - Add `lifecycleEvents: PhalanxEvent[]` to `MatchInstance`.
   - Append `match.created` event in the match constructor / `createMatch`.
   - Append `player.joined` in `joinMatch` and for bot creation.
   - Append `game.initialized` after the `system:init` `handleAction` resolves.
   - Append `game.completed` when `match.state.phase === 'gameOver'`.

4. Add `buildMatchEventLog(match: MatchInstance): MatchEventLog`:
   - Concatenate `match.lifecycleEvents` with all turn events derived from
     `match.state.transactionLog` (using `deriveEventsFromEntry`).
   - Compute `fingerprint` using `computeStateHash` (already in shared).
   - Return `MatchEventLog`.

5. Write server tests covering the full lifecycle event sequence for both a
   human-vs-human match and a bot match.

## Implementation Notes

<!-- SECTION:NOTES:BEGIN -->
- Added `lifecycleEvents: PhalanxEvent[]` to `MatchInstance` interface.
- Lifecycle event IDs use a deterministic `${matchId}:lc:${kind}` pattern to
  avoid collisions with turn events and make fingerprints reproducible.
- `rngSeed` is omitted from the `match.created` payload (fairness concern for
  live matches; seed can be revealed post-game if needed).
- `game.completed` deduplication guard lives in a private `maybeEmitGameCompleted`
  helper to keep `handleAction` under the ESLint complexity budget (20 max).
- `buildMatchEventLog` exported from `server/src/match.ts`; uses
  `computeStateHash(events[])` from `@phalanxduel/shared/hash` for the fingerprint.
- `MatchEventLogSchema` + `MatchEventLog` type added to shared; `types.ts` and
  `json-schema/MatchEventLog.json` regenerated via `pnpm schema:gen`.
- 4 new TelemetryName constants: `EVENT_MATCH_CREATED`, `EVENT_PLAYER_JOINED`,
  `EVENT_GAME_INITIALIZED`, `EVENT_GAME_COMPLETED`.
- Key lesson: shared package `dist/` must be rebuilt (`pnpm --filter @phalanxduel/shared build`)
  before running server tests whenever `telemetry.ts` or `schema.ts` changes.

## Risks and Unknowns

- `params` in `match.created` must not include `rngSeed` in plaintext (fairness
  concern in live matches). Either omit or include a masked/hashed version. The
  seed can be revealed in `game.completed` after the match ends.
- Bot player IDs are synthetic UUIDs — the `player.joined` event for a bot
  should clearly set `isBot: true` and omit personally identifying data.
- Reconnecting players receive the current state but not the event log history
  via WS. History retrieval is a persistence concern (TASK-45.4).
- The `durationMs` in `game.completed` is wall-clock time from `match.createdAt`
  to game-over. This is informational, not deterministic.
